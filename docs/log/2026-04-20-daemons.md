# Daemons

Add a `[daemons.<name>]` config section for sidecar processes that run in
parallel with the main shell. Motivating case: `dockerd` inside the
sandbox, which has to start before the user's shell and needs a clean
shutdown path when the shell exits.

### Scope

- Per-daemon: command, cwd, env (with `${VAR}` expansion), graceful-stop
  signal, shutdown timeout, restart policy (`always` / `on-failure`),
  max restart attempts, and a per-daemon `harden` override.
- Stdout/stderr go to `/airlock/daemons/<name>/(stdout|stderr).log`
  (truncated on sandbox start, appended across auto-restarts).
- Linear `attempt_number * 1s` backoff between restarts. No
  exponential backoff, no dependency ordering, no cgroup limits.
- Shutdown UI: one indicatif spinner per daemon; each row finishes with
  `shut down` or `killed` depending on whether `SIGKILL` had to be used.

### Design choices

- **Piggyback on the existing `Supervisor.start` RPC** ŌĆö daemons are
  passed as a `List(DaemonSpec)` on the start params. The alternative
  was a separate `startDaemons` method; but daemons conceptually start
  at the same point in the boot sequence as the main shell, and sharing
  the RPC keeps the contract "one call brings the sandbox up".
- **Polled `pollDaemons @6` instead of a push interface.** A streaming
  interface would push state changes over an event channel, but the
  host only cares about terminal states during a narrow shutdown
  window. A 100 ms poll during that window is trivially cheap and
  avoids adding a long-lived subscription to the supervisor schema.
- **`shutdownDaemons @7` is fire-and-forget.** Returning a completion
  signal would force the supervisor to track a "fully drained" state
  distinct from "all terminal" ŌĆö instead the host drives the spinners
  from `pollDaemons` results until all rows are terminal.
- **`DaemonSet` lives in the supervisor and is owned by the init
  callback, not the RPC server.** The init callback is the only place
  with access to the resolved `uid`/`gid` and it runs after
  `init::setup` has created `/mnt/overlay/rootfs/airlock/daemons/`.
  The RPC server holds an `Rc<RefCell<Option<DaemonSet>>>` slot that
  init populates once the set is ready; `pollDaemons` / `shutdownDaemons`
  resolve through that slot.
- **Restart loop runs regardless of first-start success.** If the
  initial spawn fails (bad binary, missing cwd) the daemon immediately
  becomes a restart attempt rather than a startup error. The main shell
  is never blocked by a misconfigured daemon.
- **Log files opened once with `truncate=true`, then `try_clone()`d per
  spawn.** Duplicated FDs share the same open file description, so the
  write offset advances across restarts ŌĆö natural append-on-restart
  semantics, truncate-on-sandbox-restart semantics, and no extra file
  handling inside the restart loop.
- **Signal is a named enum (`SIGTERM`, `SIGINT`, ŌĆ”), not an arbitrary
  integer.** Keeps the config validatable at load time and avoids the
  trap of a user writing `signal = 9` in a platform-specific way. The
  allowed set is the subset that makes sense for graceful shutdown.

### Shared process spawn path

`airlockd::process` had two nearly-identical pre-exec hooks (chroot,
chdir, setgid, setuid, optional hardening). Extracted the shared logic
into a `build_pre_exec` helper so `spawn_user` (main shell + exec) and
`spawn_daemon` both reuse it. Differences between the two:

- `spawn_user` wires stdio to a PTY and writes a diagnostic pipe for
  post-fork errors surfaced to the host.
- `spawn_daemon` redirects stdio to the pre-opened log files
  (`Stdio::from(file.try_clone())`) and uses `/dev/null` for stdin.

### Module layout

- `airlockd/src/daemon.rs` ŌĆö owns `DaemonSpec`, `DaemonSet`,
  `RestartPolicy`, `DaemonState`, the per-daemon restart-loop task,
  `graceful_stop`, and the capnp glue (`DaemonSpec::from_capnp`,
  `parse_specs`, `write_status_list`).
- `airlock-cli/src/daemon.rs` ŌĆö owns `build_specs` (config ŌåÆ wire
  format with env expansion), `run_shutdown` (indicatif spinner UI),
  and `print_verbose` (the `--verbose` daemon block).

Keeping all daemon-specific code in its own module on each side of the
RPC leaves `rpc.rs` and `cmd_start.rs` as thin orchestration layers.

### Out of scope

- Daemon-to-daemon startup ordering / dependencies.
- Health checks beyond "process alive".
- Resource limits (cgroup CPU/mem quotas).
- Log rotation / size caps.
- Reload-on-SIGHUP from host.
- Exponential backoff.
- Per-daemon uid/gid override.
